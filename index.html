Introduction

The Agile approach to project management attacks most of the common issues among software development methodologies by enhacing communication with the stakeholders, and concentrating efforts on the application's behavior instead of implementation.

Stakeholders include nearly everyone: customers, developers, managers, operators, ...

Agile requires some of the following patterns:

  • User stories: to elicit functional requirements.
  • Low-fidelity (Lo-Fi) user interfaces and storyboards to elicit UI requirements.
  • Points: to turn user stories into cost estimates.
  • Velocity: to measure ad estimate schedule.
  • Transforming user stories to acceptance test using tools like Cucumber.
  • Using Pivotal Tracker to track project progress, to calculate velocity, and to estimate time to milestones.

BDD & User Stories

Behavior-Driven Design asks questions about the behavior of an application before and during development so that the stakeholders are less likely to miscommunicate. Requirements are written down continuously refined to ensure the resulting software meets hte stakeholder's desires.

The agile version of requirements is user stories, which describe how the application is expeted to be used.

User stories are a lightweight version of requirements and replace design documents in plan-and-document methodologies. User stories also help the stakeholders to plan and prioritize development.

Combined working closely with the stakeholders and concentrating on the the application's behavior rather than the implementation, makes easier to spot missunderstandings.

Example of a User Storie (It can be written in a 3x5 inch paper card):

story1.feature 
                                
Feature: Add a book to AwesomeBooks
  As a book reader
  So that I can share a book with other book readers
  I want to add a book to AwesomeBooks database
Format 
Feature: Feature name
  As a [kind of stakeholder]
  So that [I can achieve some goal]
  I want to [do some task]

Points Velocity and Pivotal Tracker

The solution to calculate a team's productivity is to rate the stories in advance according to the difficulty, and then calculate the average of points per iteration in order to decide how many stories try to implement per iteration. This is called velocity.

  • 1. For straightforward stories.
  • 2. For medium stories.
  • 3. For very complex stories.

As long as the team rates the user stories consistently, it does not matter the iteration's punctuation.

The purpose of velicity is to provide the stakeholders an estimation of how long it will take to integrate the desired features.

User stories should not be complex that there is no clear way to approach them. If so, you should go back and evaluate with stakeholders if user storie refactor is needed.

SMART User Stories

SMART provides a helpful acronym to write good user stories.

Specific:
Feature: User can search for a movie (vague)
Feature: User can search for a movie by title (specific)

Measurable:


Feature:  RottenPotatoes should have good response time (unmeasurable)


Feature:  When adding a movie, 99% of Add Movie pages 
          should appear within 3 seconds (measurable)

Achievable:

Ideally, you should be able to implement the user story in one iteration.

Relevant:

It should add business value to one or more stakeholders. to drill down to real business value, you can keep asking "Why". take the example of adding a Facebook linking feature.
1. Why add the Facebook feature? As box office manager, I think more people will go with friends and enjoy the show more.
2. Why does it matter if they ejoy the show more? I think we will sell more tickets.
3. Why do you want to sell more tickets? Because then the theater makes more money.
4. Why does theater want to make more money? We want to make more money so that we don't go out of buiness.
5. Why does it matter that theater is in business next year? If not, I have no job.

Timeboxed:

In agile software development, a timebox is a defined period of time during which a task must be accomplished. Timeboxes are commonly used to manage software development risk. Development teams are repeatedly tasked with producing a releasable improvement to software, timeboxed to a specific number of weeks.
